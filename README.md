# Post-Quantum X.509 Assurance

Repository accompanying the paper:

`From Public-Key Linting to Operational Post-Quantum X.509 Assurance for ML-KEM and ML-DSA: Registry-Driven Policy, Mutation-Based Evaluation, and Import Validation`

This repository contains the executable artifact for registry-driven operational
post-quantum X.509 assurance over final ML-KEM and ML-DSA encodings in PKIX.

The artifact is organized around operator workflow. It assigns each executable requirement to an owner, a stage, and a
gate-pack, and emits operator-facing outputs for both CA pre-issuance and
importer-side private-key validation.

In concrete terms, the repository generates:

- CA-facing outputs for `certificate/profile` and `SPKI/public-key` gates,
  including mode-aware policy summaries, coverage summaries, gate-pack matrices,
  and operator playbooks.
- importer-facing outputs for `private-key-container/import`, including
  import-validation summaries, private-key coverage, and owner/stage workflow
  reports.
- supporting evidence artifacts for the frozen baseline comparison and the
  public appendix.

## Scope

- algorithms: `ML-KEM`, `ML-DSA`
- profile: `pkix-core`
- executable artifact surfaces:
  - `certificate/profile`

  - `ca-preissuance`

- runtime boundary:
  - `runtime-consumer` remains explicit as a boundary, but is not part of the
    executable artifact in this repository snapshot

## Operator Workflow Thesis

The core value of the artifact is not only that it lints post-quantum X.509
objects. It turns final normative requirements into owner-bound, stage-bound,
replayable decisions.

Each requirement in `requirements.json` is assigned to:

- an owner
- an execution stage
- a gate-pack
- a policy action in `strict` and `deployable` mode

This lets the repository answer three operational questions directly:

1. what a CA must block or warn on before issuance
2. what an importer must validate before accepting a private-key container
3. which outputs correspond to each gate in each policy mode

## Repository Layout

- `src/`: Python implementation
- `tests/`: regression tests
- `experiments/`: reproducible runners and replay scripts
- `corpus/`: controlled corpus, bounded appendix corpus, and manifests
- `results/`: frozen outputs used by the paper and replay path
- `docs/`: operator-facing notes and generated workflow summaries
- `reference/`: compact local reference set for normative PDFs and frozen
  methodological context
- `third_party/`: canonical vendored upstream source snapshots used by replay
  paths
- `tools/`: optional bridge tooling used for importer-side consistency checks

## Quick Start

Run the local regression suite:

```sh
./experiments/run_smoke_tests.sh
```

Run the local registry and corpus checks:

```sh
./experiments/run_extended.sh --mode strict
./experiments/run_extended.sh --mode deployable
./experiments/run_coverage.sh --mode strict
./experiments/run_coverage.sh --mode deployable
./experiments/run_private_key_coverage.sh --mode strict
./experiments/run_private_key_coverage.sh --mode deployable
```

Emit the operator workflow outputs:

```sh
./experiments/run_reference_workflow.sh
./experiments/run_operator_gate_packs.sh
./experiments/run_cross_tool_behavior.sh
```

Build the optional importer bridge when the bridge source is present:

```sh
./experiments/build_libcrux_import_check.sh
```

Run the end-to-end replay path:

```sh
./experiments/replay_freeze.sh
```

## Upstream Inputs and Appendix Replay

The replay path may rely on pinned upstream inputs and compact normative
references.

- `reference/` stores the local PDFs and frozen reference material used to
  anchor the registry and appendix narrative.
- `third_party/` is the canonical working location for vendored upstream
  sources such as `jzlint-main`, `libcrux-main`, and, where present,
  `pqc-certificates-main`.
- `tools/libcrux_import_check/` contains the small bridge used for
  importer-side seed/expanded consistency checks against `libcrux`.

The public branch keeps the upstream source snapshots in `third_party/` rather
than duplicating the same inputs as ZIP archives alongside the normative
references.

When the larger `pqc-certificates` snapshot is unavailable, the repository can still reuse the frozen appendix outputs that accompany this repository snapshot.

## Notes

- `deployable` is the default operator mode for low-noise gating.
- `strict` is the assurance-maximizing reference mode.
- In the current registry, the only non-blocking deployable requirement is
  `MLKEM-SPKI-ENCODE-DECODE-IDENTITY`.
